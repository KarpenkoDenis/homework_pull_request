## Основные команды

1. > git log –graph 

    выводит commit (сохранения) с дополнительным графическим отображением в терминале branch (веток)

2. > git clone url_repo

    Клонирует (скачивает полностью) удалённый репозиторий. 
    
    Важная особенность! Репозиторий будет скачен во внутрь вашей текущей папки. Для дальнейшей работы с клонированным репозиторием, необходимо перейти в терминале в данный репозиторий (например при помощи команды cd folder_name)

3. > git pull
    
    Скачивает commit-ы удалённого репозитория в локальный репозиторий. При этом происходит не только клонирование данных но и merge (соеденение) данных.

    > git pull remote branch
    >>git pull origin master

    >>>reomote - как правило носит название origin (это имя удалённого репозитория на GitHub)
    branch - название ветки с которой необходимо установить связь между локальным и удалённым репозиторием 

    Производит соединение локальной и удалённого репозитория по необходимой ветке
    
    > git pull --rebase remote branch
    >>git pull --rebase origin master
    >>> --rebase - вспомогательная команда которая добовляет commit-ы последовательно

    команда которая позволяет скачать commit-ы удалённого репозитория последовательно, без merge. Необходима в случае если обычный pull не проходит, например, из за того, что в удалённом репозитории есть файлы которых нет в локальном или в иных ситуациях.

4. > git push

    Отправляет commit-ы на удалённый репозиторий

    > git push --set-upstream remote branch

    >>git push --set-upstream origin master

    >>> reomote - как правило носит название origin (это имя удалённого репозитория на GitHub)
    branch - название ветки с которой необходимо установить связь между локальным и удалённым репозиторием 

    Отправляет commit-ы c выставлением связей локальный и удалённых веток

5. > git remote

    проверка названия удалённого репозитория GitHub

    > git remote add origin url_repo
    >> вместо origin может быть добавлен любое название.
    создание названия удалённого репозитория (не путать с названием папок и файлов, речь о названии для ветвлений)

3. ### git remote

    Или отправляем свою локальную работу на GitHub

    - локальный репозиторий
    - git push
    - ответ, что вообще-то у нас локальный репозиторий ни с чем не связанный
    - git remote add origin url repo - связываем
    
        - то есть origin это задаваемое имя удалённого репозитория add на GiHub - можно задать любое, стандарт origin
    - git remote - проверям что origin создалось
    - опять git push
    - получаем подсказку в виде команды git push —set-upstream remote branch, что нужно назвать ветку, которая будет соединена с нашей локальной веткой
    - git push  —set-upstream origin master - Это выставление связей веток
        - origin - название удалённого репозитория
        - master - удалённая ветка с которой мы связываем текущую ветку
    

    > Фича такого способо добавления через git remote заключается в том, что commit-ы подтянутся на GitHub с оригинальными тэгами времени создания и GitHub покажет когда commit-ы создавались.

4. ## git remote в рабочий repo

    отправляем свой репозиторий в существующий со своими файлами на GitHub

    - git remote add origin url repo - связываем
    - git remote add origin url repo
        - то есть origin это задаваемое имя удалённого репозитория add на GiHub - можно задать любое, стандарт origin
    - git remote - проверям что origin создалось
    - опять git push
        - получаем подсказку в виде команды git push —set-upstream <remote> <branch>, что нужно назвать ветку, которая будет соединена с нашей локальной веткой
    - git push  —set-upstream origin master
        - origin - название удалённого репозитория
        - master - удалённая ветка с которой мы связываем текущую ветку
    
    НО ОШИБКА!
    
    Проблема в том, что там не пусто и наш локальный репозиторий не совпадает с удалённым.
    
    Нужно в начале взять инфу оттуда (pull) соеденить (merge) и только потом отправлять (push)
    
    - git pull
    - ругается, нет связей веток
    - git pull remote branch - вытаскиваем информацию с выставлением связей веток
        - git pull origin master
    
    ОШИБКА! 
    
    Проблема в том, что репозитории слишком разные, а команда pull помимо того что вытаскивает информацию она её пытается ещё и соединить (merge)
    
    Решение - не делать merge при помощи добавления ——
    
    - git pull --rebase  remote branch - вытаскиваем информацию с такого то репо такой то ветки последовательно
        - git pull —rebase origin master
            - —rebase - команда, которая добавляет commit в текущий локальный репозиторий последовательно
        
    > Тут стоит обратить внимание, что fatal error может возникнуть даже если речь идёт о разных файлах в репозитории (то есть в репозитории уже есть какие-то файлы, а вы туда хотите добавить просто новый свой. Дак вот изначально pull не только стаскивает информацию но и пытается сделать merge, а сделать merge двух разных файлов, например, ну совсем ни как. Вот и ошибка. Поэтому в случае добавления чего либо в уже существующий репозиторий с файлами, с которым вы не были связаны, нового файла из локального репозитория, действуем через —rebase ;)
